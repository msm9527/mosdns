# 缓存系统重构与优化计划

## 1. 文档定位

本文档用于指导后续缓存系统的工程化改造，目标是把当前已有实现演进为“可恢复、可观测、可治理”的缓存体系。

需要明确两点：

- 本文档描述的是“推荐实施路线”，不是对当前代码能力的宣传页。
- 方案必须以当前仓库代码为准，优先解决已经暴露的可靠性和运维问题，再推进更高风险的架构升级。

本次规划聚焦以下范围：

- 持久化可靠性
- 启动恢复能力
- 指标与运维接口补强
- 多实例配置收敛
- 主动治理能力

本次规划暂不直接包含以下动作：

- 一步到位替换现有 L1/L2 缓存结构
- 未经验证就引入全局 `CacheManager` 作为强依赖
- 为了“设计完整”而提前实现高风险特性

## 2. 当前实现基线（基于代码）

### 2.1 插件层：`plugin/executable/cache/cache.go`

当前缓存插件已经具备以下能力：

- 两级缓存路径
  - L1：进程内热点缓存，固定 256 分片，单分片使用近似 CLOCK/FIFO 的环形淘汰
  - L2：基于 `pkg/cache.Cache` 的主缓存存储
- 命中与回源逻辑
  - 正常命中返回缓存结果
  - 支持 `lazy_cache_ttl`，过期命中后可返回陈旧数据并异步刷新
  - 异步刷新链路已使用 `singleflight` 去重，并有限流保护
- 持久化能力
  - 启动时自动执行 `loadDump`
  - 周期性 `dumpCache`
  - 关闭插件时会触发一次 `dumpCache`
  - dump 内容使用 `gzip + protobuf block` 序列化
- 运维 API
  - `GET /flush`：清空 L1/L2，并异步触发一次 dump
  - `GET /dump`：将当前快照写到响应流
  - `GET /save`：将快照写入配置的 `dump_file`
  - `POST /load_dump`：从请求体读取 dump 并加载
  - `GET /show`：按关键字分页展示缓存内容
- 当前指标
  - `cache_query_total`
  - `cache_hit_total`
  - `cache_lazy_hit_total`
  - `cache_size`

### 2.2 底层缓存层：`pkg/cache/cache.go`

底层缓存容器当前特征：

- 基于并发 map + LRU 结构封装
- 支持 `Get`、`Store`、`Range`、`Flush`、`Len`、`Close`
- 通过后台 `gcLoop` 定期清理过期项
- 读路径对过期项会即时删除

当前底层层面尚不具备：

- WAL
- 内存水位控制
- namespace 配额与优先级
- 主动刷新调度
- 持久化元数据校验

### 2.3 配置层：`config/sub_config/cache.yaml`

当前配置中已经存在多个独立 cache 实例，例如：

- `cache_all`
- `cache_all_noleak`
- `cache_cn`
- `cache_google`
- `cache_google_node`
- `cache_node`
- `cache_cnmihomo`

这说明当前体系本质上是“多个插件实例分别治理”，而不是“一个统一的全局缓存控制器”。因此，后续设计必须优先考虑兼容现有实例化方式。

## 3. 主要问题矩阵

| 问题 | 当前现状 | 影响 | 优先级 |
|------|----------|------|--------|
| dump 写盘原子性不足 | 当前 `dumpCache()` 直接 `os.Create(dump_file)` 覆盖写入 | 异常退出时可能留下不完整快照 | P0 |
| 恢复模型只有全量快照 | 启动恢复依赖单个 dump 文件，没有增量日志 | 快照窗口内变更丢失，恢复可控性差 | P0 |
| 指标维度不足 | 只有查询/命中/陈旧命中/尺寸等少量指标 | 无法判断 dump、load、lazy update、错误率与恢复耗时 | P1 |
| 管理接口偏手工 | 已有 `/flush`、`/save`、`/load_dump`、`/show`，但缺少聚合统计接口 | 运维排障与容量评估成本高 | P1 |
| 多实例策略分散 | 多个 cache 实例在 YAML 中独立配置 | 参数漂移、治理策略难统一 | P1 |
| 主动治理能力不足 | 只有 TTL 清理和被动 dump，没有内存水位、配额回收、主动刷新 | 长期运行稳定性依赖人工调参 | P2 |

## 4. 设计目标与约束

### 4.1 目标

- 提高 dump/load 的可靠性和故障恢复确定性
- 建立可量化的指标和基线，避免“改完了但无法证明更好”
- 在不破坏现有热路径的前提下，逐步补齐恢复与治理能力
- 降低多 cache 实例的配置分散和维护成本
- 为后续主动刷新、配额管理和自动治理预留扩展位

### 4.2 约束

- 兼容现有 cache 插件实例化方式与主要 API
- 优先保护命中路径，不在前两阶段引入高侵入式大改
- 新能力默认可开关，支持灰度和回滚
- 所有阶段都必须附带明确验收项和回滚路径

### 4.3 关键原则

- 先加固现有快照链路，再讨论 WAL 和统一治理
- 先补观测，再做复杂治理；没有指标的优化不可控
- 先提炼共享策略组件，再评估是否真的需要全局 `CacheManager`

## 5. 推荐技术路线

### 5.1 近期路线

不建议一开始就把“全局 `CacheManager`”作为核心交付物。更稳妥的顺序是：

1. 加固现有 snapshot 持久化链路
2. 补齐指标、统计与压测基线
3. 在 snapshot 稳定后增加 WAL
4. 再收敛配置与治理逻辑
5. 最后推进主动刷新和自动治理

这样做的原因是：当前最明确的风险点在“快照可靠性”和“缺少观测”，而不是“必须马上统一所有实例”。

### 5.2 目标架构演进

建议把能力拆成几个可独立演进的部件，而不是一步到位重写缓存插件：

- `Cache Runtime`
  - 保留现有查询热路径
  - 继续承担 L1/L2 访问、lazy update、API 暴露
- `Persistence Controller`
  - 负责 snapshot 写入、校验、恢复、WAL 回放
- `Metrics / Stats Collector`
  - 负责 Prometheus 指标和 `/stats` 聚合视图
- `Governance Hooks`
  - 负责内存水位、配额清理、主动刷新调度等可选能力
- `Shared Policy Config`
  - 负责多实例共享的默认策略，而不是强行接管实例生命周期

`CacheManager` 可以作为后续演进方向，但应满足两个前置条件后再引入：

- 已证明单实例公共策略抽取无法满足统一治理需求
- 已有完整指标支撑跨实例配额与调度决策

## 6. 分阶段实施计划

### Phase 1：快照链路加固（优先级最高）

**目标**：在不改变缓存主结构的前提下，先把现有 dump/load 做稳。

**改动重点**：

- dump 写盘改为：临时文件写入 -> `fsync` -> 原子 `rename`
- 为 dump 引入基础元信息
  - 格式版本
  - 创建时间
  - 条目数
  - 可选校验和
- load 时增加更清晰的错误分类
  - 文件不存在
  - 格式不兼容
  - 数据损坏
  - 部分读取失败
- dump / load 结果补日志字段，便于排障
- 为 dump/load/flush 增补单元测试和回归测试

**建议涉及文件**：

- `plugin/executable/cache/cache.go`
- 新增持久化辅助文件（如需要）
- 对应测试文件

**交付物**：

- 原子 dump 实现
- dump 格式元信息说明
- dump/load/flush 回归用例

**阶段验收**：

- 非正常中断不会把已有 dump 覆盖成空文件或截断文件
- 损坏 dump 不会导致进程不可启动
- `save -> load_dump -> show` 的行为可稳定回归

**回滚策略**：

- 保留旧 dump 解析兼容
- 新写盘路径可通过开关退回旧实现

### Phase 2：可观测性与性能基线

**目标**：让缓存系统的优化可测量、可回归。

**改动重点**：

- 补齐指标
  - `cache_dump_total`
  - `cache_dump_fail_total`
  - `cache_dump_duration_seconds`
  - `cache_load_total`
  - `cache_load_fail_total`
  - `cache_load_duration_seconds`
  - `cache_l1_hit_total`
  - `cache_l2_hit_total`
  - `cache_lazy_update_total`
  - `cache_lazy_update_drop_total`
- 提供 `GET /stats`
  - 输出实例级命中、容量、更新时间、dump/load 状态
- 建立压测与恢复基线
  - 命中路径 QPS/P95
  - 启动恢复时长
  - dump 耗时与文件大小

**交付物**：

- Prometheus 指标清单
- `/stats` 接口定义
- 压测脚本或基线记录文档

**阶段验收**：

- 能区分 L1/L2 命中贡献
- dump/load 错误与耗时可观测
- 至少形成一组可复用的基线结果，用于后续阶段回归

**回滚策略**：

- 新指标和 `/stats` 为增量能力，关闭即可回退

### Phase 3：恢复模型升级为 Snapshot + WAL

**目标**：把“只能恢复到最近一次全量快照”升级为“快照 + 增量回放”。

**改动重点**：

- 定义 WAL 记录模型
  - `store`
  - `delete`
  - `flush`
  - 可选 `expire`/`compact` 标记
- 启动恢复流程调整为
  - 先加载 snapshot
  - 再回放 WAL
  - WAL 损坏时按策略中止、跳过或截断
- 定义日志滚动和截断规则
  - dump 成功后生成新的 checkpoint
  - 旧 WAL 可归档或删除
- 为 WAL 增加格式版本和校验

**设计约束**：

- WAL 必须可禁用
- WAL 写入不能明显拖慢热路径
- 回放失败必须有明确降级策略，至少能退回到 snapshot 启动

**交付物**：

- WAL 格式说明
- snapshot + WAL 恢复流程说明
- 故障场景测试用例

**阶段验收**：

- 正常启停下，最近一次 checkpoint 之后的增量可恢复
- WAL 损坏时服务仍可按降级路径启动
- 热路径性能回归在可接受范围内

**回滚策略**：

- 通过配置切回“仅 snapshot”模式

### Phase 4：配置收敛与共享策略组件

**目标**：降低多实例配置漂移，但不强行引入全局统一调度器。

**改动重点**：

- 提炼共享配置项或配置模板
  - dump 策略
  - 持久化策略
  - 刷新策略
  - 治理阈值
- 将实例公共逻辑抽为共享组件
  - persistence policy
  - metrics policy
  - governance policy
- 评估是否需要引入轻量级实例注册表
  - 仅用于聚合状态和统一统计
  - 不直接接管实例查询热路径

**不建议在本阶段默认承诺的内容**：

- 直接把所有 cache 实例迁移到一个强耦合的全局 `CacheManager`
- 在未完成 Phase 2 指标补齐前做跨实例配额决策

**交付物**：

- 共享配置方案
- 实例公共策略抽象
- 兼容旧配置的迁移说明

**阶段验收**：

- 典型 cache 实例可以减少重复配置
- 旧配置仍可运行
- 聚合统计不影响现有请求链路

**回滚策略**：

- 保留旧配置解析路径
- 新共享策略层可逐实例灰度启用

### Phase 5：主动治理与自动运维

**目标**：补齐长期运行能力，而不再依赖人工频繁干预。

**改动重点**：

- 主动刷新调度器
  - 去重
  - 并发上限
  - 限速
  - 热点名单批量刷新
- 内存水位治理
  - 高水位触发清理
  - 低水位停止清理
- namespace 配额与优先级回收
- 管理接口扩展
  - `POST /refresh`
  - `POST /compact`

**交付物**：

- 主动刷新调度设计
- 内存治理阈值策略
- 运维接口补充说明

**阶段验收**：

- 可在无外部请求触发的情况下执行主动刷新
- 达到高水位时有可观测的自动治理行为
- 清理策略对热点命中影响可量化

**回滚策略**：

- 主动治理能力默认开关控制
- 水位和配额策略支持逐实例关闭

## 7. 验收标准

### 7.1 功能验收

- dump/save/load/flush/show 行为与文档一致
- 新增 `/stats`、`/refresh`、`/compact` 时，接口语义明确且可回归
- Snapshot + WAL 模式下，恢复流程可重复验证

### 7.2 性能验收

- 命中路径 P95 不得明显劣化
  - 建议以 Phase 2 建立的基线为准，默认回归阈值不超过 5%
- dump 和恢复耗时有稳定基线
- WAL 开启后写入成本可观测，且在可接受范围内

### 7.3 稳定性验收

- 异常退出后不会因为 dump 文件损坏导致服务不可用
- WAL 回放失败存在可执行的降级路径
- 长时运行下，内存水位治理与清理行为符合预期

### 7.4 运维验收

- 关键指标可接入 Prometheus 并支撑报警
- 运维接口覆盖“查看、保存、恢复、刷新、整理”核心场景
- 故障排查时可明确判断问题属于命中路径、持久化链路还是治理链路

## 8. 风险与应对

| 风险 | 说明 | 应对 |
|------|------|------|
| WAL 带来额外 IO 成本 | 高频写入场景下可能放大写盘压力 | 批量写、刷盘策略可配置、默认可关闭 |
| 过早抽象统一管理层 | 改动面大，容易影响现有多实例行为 | 先抽共享策略组件，再决定是否升级为统一管理器 |
| 指标设计不完整 | 后续治理建立在错误观测上会放大问题 | 先完成 Phase 2 再推进高阶治理 |
| 主动清理误伤热点 | 命中率可能波动 | 配额分级、热点保护、可回滚参数 |
| 恢复链路复杂度上升 | snapshot + WAL 容易引入兼容问题 | 保留纯 snapshot 回退模式，逐步灰度 |

## 9. 建议执行顺序

建议严格按以下顺序推进：

`Phase 1 -> Phase 2 -> Phase 3 -> Phase 4 -> Phase 5`

原因如下：

- 如果 Phase 1 没做稳，后续所有恢复增强都建立在脆弱快照链路上
- 如果 Phase 2 没补齐，Phase 3 以后很难判断收益和回归风险
- 如果在 Phase 3 前直接推进 Phase 4/5，会把“架构复杂度”放在“可靠性和可观测性”之前

## 10. 近期建议落地清单

如果要开始真正实施，建议先只做下面这批低风险项：

- dump 原子提交
- dump/load 错误分类与日志增强
- dump/load/flush 回归测试
- dump/load 关键指标补齐
- `GET /stats` 只读接口

这批工作完成后，再决定是否进入 WAL 设计与多实例策略收敛阶段。
